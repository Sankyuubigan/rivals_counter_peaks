# Спецификация архитектуры: OTA-обновления (Over-The-Air)

В этом документе описана логика работы системы динамических обновлений баз данных (stats.json) и изображений (иконки героев и карты) без необходимости перевыпуска расширения в магазине Overwolf.

## 1. Система баз данных (JSON)

Проблема: Ранее для обновления меты и контрпиков требовалось собирать новый билд расширения и отправлять его на модерацию в Overwolf. 
Решение: Приложение умеет скачивать свежие базы напрямую с GitHub и хранить их в `localStorage`.

### Как это работает под капотом:
1. **Дефолтная база из коробки:**
   * В файле `overwolf_app/database/game_entities_dict.json` прописан ключ `default_db_file` (например, `marvel_rivals_stats_20250221-033306.json`).
   * Этот файл **всегда** физически лежит в папке `overwolf_app/database/stats/` внутри установленного расширения. Он используется как надежный "фоллбэк", если у юзера нет интернета или он ничего не скачивал.

2. **Логика загрузки (`logic.js` -> `init()`):**
   * При старте скрипт проверяет `localStorage.getItem('active_db_name')`.
   * Если там пусто или стоит `local`, скрипт грузит базу из коробки.
   * Если там указано имя файла (например, `marvel_rivals_stats_20260601.json`), скрипт берет содержимое этой базы из кэша `localStorage.getItem('saved_dbs')`.

3. **Синхронизация с GitHub (`desktop.js`):**
   * При нажатии "Проверить папку на GitHub", приложение делает запрос к **GitHub REST API**:
     `https://api.github.com/repos/Sankyuubigan/rivals_counter_peaks/contents/overwolf_app/database/stats?ref=master`
   * API возвращает список файлов. Приложение фильтрует только `.json` файлы.
   * Юзер нажимает "Скачать" -> файл скачивается, парсится в объект и сохраняется в `saved_dbs` внутри `localStorage`.

### Воркфлоу разработчика (как выпустить апдейт):
1. Сгенерировать новый `stats.json` с датой в названии.
2. Загрузить его на GitHub в папку `master/overwolf_app/database/stats/`.
3. **Всё.** Юзеры увидят апдейт по кнопке. Обновлять само расширение не нужно.

## 2. Динамическая подгрузка изображений (Иконки и Карты)

Проблема: Overwolf не может доставать картинки напрямую из файлов игры. При добавлении новых героев/карт в базу (через OTA), их картинок физически не будет в папке расширения у пользователя.
Решение: Трехступенчатый Fallback-механизм (Локал -> GitHub -> Текстовая заглушка).

### Как это работает под капотом (`desktop.js` и `in_game.js`):
Используются функции `applyHeroImage` и `applyMapImage`.

1. **Шаг 1 (Локально):** Скрипт пытается применить картинку из локальной папки `../../resources/heroes_icons/название_героя_1.png`.
2. **Шаг 2 (GitHub):** Если картинки нет (срабатывает событие `onerror` у объекта `Image`), скрипт подставляет URL напрямую с GitHub:
   `https://raw.githubusercontent.com/.../master/overwolf_app/resources/heroes_icons/название_героя_1.png`
   Браузер скачивает картинку и кэширует её на жесткий диск.
3. **Шаг 3 (Заглушка):** Если на GitHub картинки тоже нет (ошибка 404), скрипт применяет стили:
   * Убирает фоновое изображение.
   * Создает `<span>` класс `.hero-fallback-text`.
   * Вписывает туда первые 5 букв имени героя белым цветом на темном фоне.
   Это гарантирует, что UI (иконки ролей, бейджи приоритета) не сломается и останется читаемым.

*Примечание:* Результат каждого шага сохраняется в объект `imageCache` в оперативной памяти (ОЗУ). Это предотвращает "моргание" интерфейса при перерисовках списка и бережет ресурсы процессора, так как `onerror` не вызывается повторно для одних и тех же картинок.

### Воркфлоу разработчика (как добавить героя):
1. Обновить базу данных (закинуть новый `.json` на GitHub).
2. Нарезать иконку героя в формате `.png` по стандартам старых иконок (нейминг: `имя_героя_1.png`).
3. Закинуть иконку на GitHub в `master/overwolf_app/resources/heroes_icons/`.
4. **Всё.** У юзеров, которые скачают новую базу, иконка подтянется сама.

## Возможные проблемы и решения (Траблшутинг)

1. **Базы с GitHub не ищутся (Ошибка в логах):**
   * *Причина:* GitHub API для неавторизованных запросов имеет лимит — **60 запросов в час с одного IP**. Если юзер будет спамить кнопку "Проверить обновления", он словит лимит.
   * *Решение:* Подождать час. Для обычного использования (раз в пару недель) этого лимита более чем достаточно.

2. **Превышение квоты `localStorage` (QuotaExceededError):**
   * *Причина:* В браузере лимит на `localStorage` около 5-10 МБ.
   * *Решение:* Интерфейс позволяет удалять старые базы. В текущем виде каждая база весит ~100-200 КБ, поэтому лимит будет достигнут не скоро (около 30-50 баз).

3. **Сломался UI у нового героя (Текст съехал):**
   * *Причина:* Скорее всего, имя героя слишком длинное.
   * *Решение:* Fallback-механизм обрезает имя до 5 символов (`heroName.substring(0, 5)`). Если этого недостаточно, можно подкрутить CSS-класс `.hero-fallback-text` (уменьшить шрифт или добавить `word-break`).
